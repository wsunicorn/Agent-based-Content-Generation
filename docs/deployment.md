# Phase 11 Deployment Notes

## Recommended target

Use Render first for this project.

Why:

- The app needs four production services: Django/Daphne web, Celery worker, Redis-compatible queue/cache, and Postgres.
- Render supports web services, background workers, Postgres, and Key Value Redis-compatible instances from one `render.yaml`.
- The repo now includes a Docker path, so the same image can run on Render, Railway, a VPS, or local Docker Compose.

Important limitation:

- Free hosting is not a good fit for local LLMs. Ollama with 7B/8B models needs CPU/RAM/disk that free web instances usually do not provide.
- For public deployment, use `LLM_PROVIDER=gemini` or point `OLLAMA_BASE_URL` to a separate machine/VPS that you control.
- A real Celery worker is required for this multi-agent pipeline. On Render, background workers are not available on the free service plan, so a fully working public deployment is "low cost", not truly free.

## Files added for deployment

- `Dockerfile` - production container image.
- `.dockerignore` - keeps local secrets and caches out of the image.
- `docker-compose.prod.yml` - local production-like stack.
- `render.yaml` - Render Blueprint for web, worker, Redis-compatible Key Value, and Postgres.
- `scripts/build.sh` - native Render/Railway-style build helper.
- `scripts/start-web.sh` - migrate, collect static, start Daphne.
- `scripts/start-worker.sh` - start Celery worker.

## Render deployment

1. Push the repo to GitHub.
2. Create a Render account and open **Blueprints**.
3. Create a new Blueprint Instance from this repository.
4. Render will read `render.yaml` and create:
   - `content-pipeline-web`
   - `content-pipeline-worker`
   - `content-pipeline-redis`
   - `content-pipeline-db`
5. When prompted, provide:
   - `GOOGLE_API_KEY` for the worker.
   - `TAVILY_API_KEY` if you want web research. Leave empty to run without web search.
6. After deploy, open the web service URL.
7. Login with Basic Auth:
   - username: `admin`
   - password: generated by Render in `APP_BASIC_AUTH_PASSWORD`.
8. Check health:
   - `/api/health/`

Suggested Render settings:

- Region: `singapore` for Vietnam latency.
- Web plan: `starter` for a usable demo.
- Worker plan: `starter`, `CELERY_WORKER_CONCURRENCY=1`.
- Postgres: `free` is okay only for temporary demo because it expires; use `basic-256mb` when you want persistence.
- Redis-compatible Key Value: free is fine for a small demo, paid if you need persistence guarantees.

## Railway alternative

Railway is also suitable if you prefer a simpler dashboard and can accept usage-based billing.

Recommended Railway shape:

- Service 1: Web
  - Dockerfile: `Dockerfile`
  - Start command: `bash scripts/start-web.sh`
- Service 2: Worker
  - Dockerfile: `Dockerfile`
  - Start command: `bash scripts/start-worker.sh`
- Plugin/service: Postgres
- Plugin/service: Redis

Set the same environment variables:

- `DJANGO_SETTINGS_MODULE=config.settings.production`
- `DEBUG=False`
- `SECRET_KEY=<strong random value>`
- `DATABASE_URL=<Railway Postgres URL>`
- `REDIS_URL=<Railway Redis URL>`
- `CACHE_URL=<Railway Redis URL>`
- `ALLOWED_HOSTS=<your Railway public host>`
- `CSRF_TRUSTED_ORIGINS=https://<your Railway public host>`
- `APP_BASIC_AUTH_ENABLED=True`
- `APP_BASIC_AUTH_USERNAME=admin`
- `APP_BASIC_AUTH_PASSWORD=<strong password>`
- `LLM_PROVIDER=gemini`
- `LLM_AGENT_PROVIDERS=`
- `LLM_AGENT_MODELS=`
- `GOOGLE_API_KEY=<your key>`
- `TAVILY_API_KEY=<optional>`

## Local production-like test

Set these in `.env` before testing:

```env
SECRET_KEY=replace-with-a-long-random-value
APP_BASIC_AUTH_ENABLED=True
APP_BASIC_AUTH_USERNAME=admin
APP_BASIC_AUTH_PASSWORD=change-me
```

Run:

```powershell
docker compose -f docker-compose.prod.yml up --build
```

Open:

```text
http://localhost:8000/
```
